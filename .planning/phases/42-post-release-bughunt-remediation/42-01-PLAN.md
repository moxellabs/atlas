# Plan 42-01: Fix CLI Runtime and Repo Command Regressions

## Goal

Make CLI command context and repo operations consistent after the 0.1.3 production regression fixes.

## Objective

Executor will add failing tests first, then fix CLI env propagation, positional duplication, and repo remove target resolution.

## Scope

- Ensure `serve` passes `context.env` into dependency loading.
- Ensure Commander positionals appear once in `CliCommandContext.argv` and `context.args`.
- Make `repo remove` use shared repo target resolution like `repo show` / `repo doctor`.
- Preserve JSON/scripted behavior and existing command aliases.

## Out of Scope

- No MCP identity/resource-prefix changes; covered in 42-02.
- No store/source adapter changes; covered in 42-03.
- No release automation changes.

## Findings From Bug Hunt

### `serve` ignores runtime env

`apps/cli/src/commands/serve.command.ts` calls:

```ts
const deps = await buildCliDependencies({
  cwd: context.cwd,
  ...(configPath === undefined ? {} : { configPath }),
});
```

It must pass `env: context.env` so setup-generated `$HOME/.moxel/atlas/config.yaml`, `ATLAS_CONFIG`, `ATLAS_IDENTITY_ROOT`, `ATLAS_CACHE_DIR`, and GHES tokens resolve consistently.

### Commander positional duplication

`apps/cli/src/index.ts` currently combines action values and `command.args`:

```ts
const positionals = [
  ...(values.filter((value) => typeof value === "string") as string[]),
  ...command.args,
];
```

Commander already includes declared args in action values. This can duplicate inputs like `['foo', 'foo']`.

### `repo remove` bypasses target resolver

`apps/cli/src/commands/repo.command.ts` uses raw `context.argv[1]`. It should resolve bare names and validate targets consistently.
